## 🔄 PROCESS (8 Phases)

### Phase 1: Context Gathering & Planning Route (Silent)
- **Kiro check first** → .kiro/specs/design.md → AUTO-PASS Gate 2
- **Brainstorm Artifacts scan (BẮT BUỘC):** Quét thư mục artifacts của session `brain/<projectId>/` để tìm và đọc nội dung tất cả các file brainstorm/thiết kế tạm (`*_brainstorm.md`, `*_review.md`, v.v.). Tích hợp toàn bộ nội dung của chúng vào context đầu vào trước khi tiến hành lập kế hoạch.
- **Claude Planner check (New):** Invoke `claude-planner` skill to delegate planning to Claude CLI or Codex CLI (gồm cả việc truyền đầy đủ context brainstorm vào prompt).
  - If Claude CLI (Fable) or Codex CLI (Sol) is installed & authenticated: Execute script to create `implementation_plan.md` via Claude Fable 5 or Codex GPT Sol (fallback).
  - If reasoning CLIs are missing/expired: Fall back immediately to planning using the current active model.
- Fallback: Read BRIEF.md → extract entities, relationships, rules
- Read .project-identity, TECH-SPEC.md, NeuralMemory

### Phase 2: Data Model Design
- Use either the plan generated by Claude (if successful) or model-generated plan.
- Per entity: Fields, Indexes, Relationships, Constraints
- Template → `templates/design-templates.md`

### Phase 3: API Contract (nếu có API)
- Per endpoint: Method, Auth, Request, Response, Notes

### Phase 4: State Machine (nếu có stateful flows)
- States, transitions, triggers, guards

### Phase 5: Self-Review Checklist
- Data integrity, performance, consistency, edge cases, security
- **Behavioral Spec Audit:** Run behavioral design check. Verify if the database schema tracks user habits, streaks, XP, or customization values, and if the API supports notifications (triggers) and single-tap actions (Fogg ability optimization) per [behavioral-design](file:///Users/trungkientn/Dev/NodeJS/main-awf/skills/behavioral-design/SKILL.md).
- Full checklist → `templates/design-templates.md`

### Phase 6: Multi-Role Architecture Review (AI Simulated)
- 5 roles: DBA, Backend Lead, Security, QA, SRE
- P0/P1 issues → fix draft before presenting

### Phase 7: Present & Approval
- Show design doc → user review → approve

### Phase 8: Write Design Doc
- Save to `docs/architecture/<feature>_design.md`
- Approval marker + NeuralMemory tag → proceed Gate 3

## 🔙 DESIGN DEVIATION PROTOCOL
Key changes in the schema must trigger a re-approval step:
1. ⛔ DỪNG CODE
2. Inform user → update design doc → re-approve
3. Update marker: "Revision 2" → proceed code
